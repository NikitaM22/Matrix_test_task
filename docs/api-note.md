# API-часть

## 1. Как должен выглядеть запрос на отправку текстового сообщения

Для отправки обычного текстового сообщения я ожидаю запрос вида:

- метод: `PUT`;
- URL: `/_matrix/client/v3/rooms/{roomId}/send/m.room.encrypted/{txnId}`;
- авторизация: bearer token в заголовке `Authorization: Bearer <access_token>`;
- тип контента: `Content-Type: application/json`.

Минимально полезное тело запроса я ожидаю таким:

```json
{
  "msgtype": "m.text",
  "body": "Hello from test"
}
```

Ключевые поля:

- `msgtype`: тип сообщения. Для обычного текста ожидается `m.text`.
- `body`: сам текст сообщения, который должен увидеть пользователь в ленте.

В реальном проекте в теле могут появляться и дополнительные поля, например форматированный текст или метаданные клиента, но для базового сценария отправки обычного сообщения достаточно `msgtype` и `body`.

## 2. Какие ответы сервера ожидаются

### Успешная отправка

При успешной отправке клиенту нужен как минимум признак того, что сообщение принято сервером. Практически я ожидаю:

- HTTP-статус `200 OK`;
- JSON-ответ с идентификатором созданного события, например:

```json
{
  "event_id": "$exampleEventId"
}
```

Этого достаточно, чтобы клиент считал сообщение принятым и мог связать локальное действие пользователя с серверным событием.

### Несуществующая комната

Если `roomId` не существует или недоступен, я ожидаю:

- HTTP-статус `404 Not Found` или близкий по смыслу код в рамках контракта;
- JSON с кодом ошибки и человекочитаемым описанием;
- отсутствие `event_id`.

Для клиента это должно означать, что сообщение не было отправлено и не должно показываться как успешно доставленное.

### Отсутствующий или невалидный токен

Если токен отсутствует, истёк или неверен, я ожидаю:

- HTTP-статус `401 Unauthorized`, а в некоторых реализациях `403 Forbidden`;
- JSON с кодом ошибки авторизации;
- отсутствие `event_id`.

Клиент должен показать пользователю понятный результат: сессия недействительна, требуется повторный вход.

## 3. Как проверить идемпотентность по txnId

Проверку идемпотентности я бы строил так:

1. Подготовить валидные `roomId`, `access_token`, `eventType = m.room.message` и тело сообщения.
2. Сгенерировать конкретный `txnId`, например `txn-001`.
3. Отправить первый `PUT`-запрос и сохранить ответ сервера, в частности `event_id`.
4. Повторить тот же запрос ещё раз без изменения `txnId`, `roomId`, `eventType` и тела.
5. Сравнить результаты двух ответов.

Ожидаемые проверки:

- второй ответ должен быть успешным, а не создавать новую сущность;
- `event_id` во втором ответе должен совпасть с первым;
- в истории комнаты должно появиться только одно сообщение с данным текстом и данным событием;
- если в клиенте включён локальный ретрай, повторная отправка после таймаута также не должна порождать дубль.
