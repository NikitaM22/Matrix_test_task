# Краткий анализ сценария

## Объект тестирования
В рамках этого задания объектом тестирования я считаю пользовательский сценарий отправки обычного текстового сообщения из веб-клиента мессенджера в существующую комнату. Граница проверки проходит через несколько этапов:

- UI веб-клиента: пользователь может открыть комнату, ввести текст, отправить сообщение и увидеть результат в ленте.
- Клиентскую логику: формирование запроса на отправку сообщения, передача авторизации, обработка успешного и ошибочного ответа, защита от дублирования при повторном `txnId`.
- API-сервер: приём запроса `PUT /_matrix/client/v3/rooms/{roomId}/send/{eventType}/{txnId}`, валидация токена, существования комнаты и возврат результата.

При этом я не углубляюсь во внутреннюю серверную реализацию Matrix и не рассматриваю доставку сообщения другим участникам комнаты как обязательную часть данного задания. Основной фокус - путь "текущий пользователь отправил сообщение и получил подтверждённый результат в своём клиенте".

## Что считать успешным результатом
Успешным я считаю не только HTTP-успех на уровне API, но и завершённый пользовательский сценарий:

- сервер принял сообщение и вернул признак успеха, например `event_id`;
- веб-клиент корректно обработал ответ;
- пользователь видит отправленное сообщение в ленте выбранной комнаты.

Если сервер ответил успешно, но сообщение не появилось в ленте, для пользователя сценарий нельзя считать полностью успешным. Это уже дефект интеграции между UI и API или проблема обновления состояния клиента.

## Основные риски и граничные случаи
Ключевые риски в этом сценарии следующие:

- сообщение уходит не в ту комнату из-за ошибки выбора комнаты или неправильного `roomId`;
- текст отображается в UI, но фактически API вернул ошибку, а клиент показал ложный успех;
- повторный запрос с тем же `txnId` создаёт дубликат вместо идемпотентного результата;
- отсутствует или истёк токен, и пользователь не получает понятную ошибку;
- комната не существует или недоступна, но клиент не обрабатывает это корректно;
- пустое сообщение, слишком длинное сообщение, сообщение из одних пробелов;
- сетевые задержки: сообщение отправлено, но пользователь не видит его сразу и воспринимает это как сбой;
- повторный клик по кнопке отправки при медленном ответе сервера может приводить к дублям.

Отдельно я бы обратил внимание на пограничное поведение вокруг идемпотентности: важно проверить не только "два запроса подряд", но и сценарий повторной отправки после таймаута/ретрая клиента, когда пользователь не уверен, ушло сообщение или нет.

## Допущения
Так как требования описаны не полностью, я делаю следующие рабочие допущения:

- для обычного текстового сообщения используется `eventType = m.room.encrypted`;
- в теле запроса передаются как минимум текст сообщения и тип сообщения, например `msgtype = m.text` и `body = "..."`;
- успешный ответ содержит идентификатор события, достаточный для того, чтобы клиент считал сообщение принятым;
- существующая комната уже доступна пользователю, либо пользователь имеет право её открыть;
- авторизация выполняется через bearer token;
- проверка доставки сообщения другим пользователям и мультидевайсная синхронизация в объём этого задания не входят;
- для E2E-автотеста достаточно подтвердить сценарий для одного пользователя в одной комнате.

Если бы это был реальный проект, перед стартом тестирования я бы отдельно уточнил требования к пустым сообщениям, ограничениям длины, локальному эху сообщения в UI и точному контракту ошибок API.
