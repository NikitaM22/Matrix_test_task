# Ручное тестирование

## Чек-лист

1. Пользователь с валидной авторизацией может открыть существующую комнату из списка.
2. После открытия комнаты отображается история сообщений и доступна форма ввода нового сообщения.
3. Поле ввода принимает обычный текст на латинице.
4. Поле ввода принимает обычный текст на кириллице.
5. Сообщение отправляется по основному пользовательскому действию: кнопка `Send` или клавиша `Enter` в зависимости от дизайна клиента.
6. После успешной отправки сообщение появляется в ленте текущей комнаты.
7. Отправленное сообщение визуально отличается как сообщение текущего пользователя или хотя бы однозначно отображается в ленте.
8. После успешной отправки поле ввода очищается или возвращается в исходное состояние, готовое к следующему сообщению.
9. При попытке отправки в несуществующую комнату пользователь получает понятную ошибку и сообщение не появляется в ленте.
10. При невалидной или истекшей авторизации отправка не выполняется, а пользователь получает понятное уведомление о необходимости заново войти.
11. Повторный запрос с тем же `txnId` не создаёт дубликат сообщения в ленте.
12. При временной задержке ответа сервера клиент не создаёт визуальные дубли из-за повторного нажатия на отправку.

## Тест-кейсы

### TC-01. Успешная отправка сообщения в существующую комнату

- Цель: проверить, что авторизованный пользователь может отправить обычное текстовое сообщение и увидеть его в ленте.
- Предусловия:
  - пользователь авторизован;
  - существует доступная пользователю комната;
  - комната открыта или может быть открыта из списка.
- Шаги:
  1. Открыть существующую комнату.
  2. Ввести в поле сообщения текст `Hello from test`.
  3. Нажать кнопку отправки или выполнить отправку через `Enter`.
- Ожидаемый результат:
  - на UI сообщение появляется в ленте текущей комнаты;
  - сообщение отображается как отправленное текущим пользователем;
  - API отправляет запрос `PUT /_matrix/client/v3/rooms/{roomId}/send/m.room.encrypted/{txnId}`;
  - сервер возвращает успешный статус и идентификатор события, например `event_id`.

### TC-02. Попытка отправки в несуществующую комнату

- Цель: проверить корректную обработку ошибки при невалидном `roomId`.
- Предусловия:
  - пользователь авторизован;
  - известен несуществующий `roomId` или подготовлен сценарий, в котором клиент отправляет запрос в удалённую/несуществующую комнату.
- Шаги:
  1. Сформировать попытку отправки сообщения в несуществующую комнату.
  2. Отправить текст `Message to missing room`.
- Ожидаемый результат:
  - на UI пользователь видит понятную ошибку отправки;
  - сообщение не фиксируется в ленте как успешно отправленное;
  - API возвращает ошибку уровня `404 Not Found` или аналогичную бизнес-ошибку "room not found";
  - повторная визуализация комнаты не должна создавать ложное сообщение.

### TC-03. Попытка отправки без действующей авторизации

- Цель: проверить, что клиент и API корректно обрабатывают отсутствие валидного токена.
- Предусловия:
  - токен отсутствует, истёк или подменён на невалидный;
  - пользователь находится в состоянии, из которого пытается отправить сообщение.
- Шаги:
  1. Открыть форму отправки сообщения.
  2. Ввести текст `Unauthorized message`.
  3. Выполнить отправку.
- Ожидаемый результат:
  - на UI пользователь получает сообщение о невалидной сессии или необходимости повторно авторизоваться;
  - сообщение не появляется в ленте как успешно отправленное;
  - API возвращает `401 Unauthorized` или `403 Forbidden` в зависимости от контракта;
  - клиент не должен маскировать ошибку как успешную отправку.

### TC-04. Повторная отправка с тем же txnId

- Цель: проверить идемпотентность API при повторном запросе.
- Предусловия:
  - пользователь авторизован;
  - есть существующая комната;
  - доступен способ повторить тот же API-запрос с тем же `txnId`.
- Шаги:
  1. Отправить сообщение `Idempotency check` с уникальным `txnId`.
  2. Повторить тот же запрос без изменения `roomId`, `eventType`, тела и `txnId`.
  3. Проверить ответ API и ленту сообщений.
- Ожидаемый результат:
  - сервер возвращает тот же результат, что и при первом запросе, например тот же `event_id`;
  - в ленте присутствует только один экземпляр сообщения;
  - повторный запрос не создаёт новое сообщение и не меняет уже созданное событие;
  - для пользователя не возникает визуального дубликата.
