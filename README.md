# Matrix Message E2E Assignment

Тестовое задание на тему проверки отправки текстового сообщения в веб-клиенте мессенджера с использованием `Playwright + JavaScript`.

## Что находится в репозитории

- `docs/scenario-analysis.md` - краткий анализ сценария, границы тестирования, риски и допущения.
- `docs/manual-testing.md` - чек-лист и 4 тест-кейса.
- `docs/api-note.md` - заметка по API отправки сообщения и проверке идемпотентности.
- `tests/message.e2e.spec.js` - один E2E-тест на Playwright (короткий orchestration-слой).
- `tests/pages/message.page.js` - Page Object Model (POM) со стабильными локаторами и основными пользовательскими действиями.
- `playwright.config.js` - минимальная конфигурация Playwright.
- `.env.example` - пример переменных окружения для запуска.

## Принятые допущения

- Отправка обычного текста выполняется через `eventType = m.room.encrypted`.
- В теле запроса достаточно полей `msgtype = m.text` и `body`.
- Успешным результатом считается не только успешный ответ API, но и отображение сообщения в ленте пользователя.
- Авторизация передаётся через bearer token, но на уровне E2E это скрыто за UI-логином.
- Тест использует адаптивный параметризуемый подход: настраивается через .env, валидирован реальным прогоном на app.element.io и запускается с локальной авторизованной сессией. Данные аккаунта и сессии в репозиторий не включаются, поэтому сценарий воспроизводим для проверяющего через npm run auth:setup.

## Почему тест сделан параметризуемым

Публичные Matrix-окружения могут быть нестабильны для E2E: меняются тексты кнопок, включается SSO, появляются промежуточные экраны или rate limit. Поэтому тест написан как демонстрация подхода:

- использует устойчивые локаторы по ролям, aria-именам, placeholder и тексту;
- ждёт признаки успешного логина и отправки сообщения через явные ожидания;
- дополнительно отслеживает сетевой `PUT`-запрос на отправку `m.room.encrypted`;
- генерирует уникальный текст сообщения

## Что я предложил бы для повышения стабильности тестов

Если бы была возможность немного улучшить само приложение, я бы предложил:

- добавить `data-testid` для поля логина, поля пароля, списка комнат, поля ввода сообщения и кнопки отправки;
- иметь отдельную тестовую комнату и тестового пользователя;
- предусмотреть seed/cleanup-хуки для комнаты;
- упростить вход на тестовом окружении без SSO и капчи;
- логировать `event_id` в UI или в ответах dev-инструментов для упрощения отладки.


## Как запустить

### 1. Установка зависимостей

```bash
npm install
```

### 2. Установка браузеров Playwright

```bash
npx playwright install chromium
```

### 3. Настройка переменных окружения

Скопировать пример:

```bash
cp .env.example .env
```

Заполнить значения:

- `BASE_URL` - URL веб-клиента;
- `LOGIN_USERNAME` - логин пользователя;
- `LOGIN_PASSWORD` - пароль пользователя;
- `ROOM_NAME` - имя существующей комнаты;
- `MESSAGE_PREFIX` - префикс для уникального текста сообщения.

### 4. Подготовка авторизованной сессии (обязательно один раз)

Тест запускается через persistent browser profile (`playwright/.user-data`), поэтому перед первым прогоном нужно один раз подготовить сессию:

```bash
PLAYWRIGHT_CHROMIUM_EXECUTABLE_PATH="/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" npm run auth:setup
```

Дальше:

1. в открывшемся браузере вручную завершить логин (включая SSO/подтверждения, если они есть);
2. вернуться в терминал и нажать `Enter`, чтобы сохранить профиль сессии.

Если сессия истекла или повреждена, повторить этот шаг.

### 5. Запуск теста

```bash
PLAYWRIGHT_CHROMIUM_EXECUTABLE_PATH="/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" npm test
```

Или в режиме headed:

```bash
PLAYWRIGHT_CHROMIUM_EXECUTABLE_PATH="/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" npm run test:e2e:headed
```

## Что именно проверяет E2E-тест

Тест:

1. открывает веб-клиент;
2. проверяет, что уже открыт авторизованный shell (логин предварительно подготавливается через `auth:setup`);
3. находит и открывает существующую комнату;
4. отправляет уникальное текстовое сообщение;
5. проверяет, что был выполнен сетевой запрос на `send/m.room.encrypted`;
6. проверяет, что сообщение видно пользователю в ленте.

## Ограничения текущей реализации

- Тест запускался на реальном публичном окружении с реальной учётной записью и существующей комнатой.
- Текущая версия сценария валидирована на `app.element.io` (`validated on app.element.io`).
- Учётные данные и сессионные артефакты в репозиторий не включаются (используются локальные `.env` и профиль `playwright/.user-data`), поэтому для повторного запуска требуется своя валидная сессия.
- Проверка идемпотентности `txnId` описана в документации и тест-кейсах, но отдельным Playwright-тестом здесь не автоматизирована, так как в задании требуется только один E2E-тест на основной пользовательский сценарий.

## Почему часть файлов не в репозитории

В `.gitignore` намеренно добавлены:

- `node_modules/` - локальные зависимости, восстанавливаются через `npm install`;
- `playwright-report/`, `test-results/` - артефакты прогонов;
- `.env` - локальные переменные и потенциально чувствительные данные;
- `playwright/.auth/`, `playwright/.user-data/` - локальная авторизационная сессия и профиль браузера.

Данные файлы будут недоступны в публичном репозитории, поэтому перед запуском НУЖНО!:

1. создать свой `.env` на основе `.env.example`;
2. выполнить `npm run auth:setup`;
3. только затем запускать `npm run test:e2e:headed` или `npm test`.

## Фактически затраченное время

Около `4 часов` на анализ, документацию, подготовку структуры проекта и написание E2E-теста.
